# Input Architecture Direction Established (System Button + Chord Model)

**Date:** 2026-04-20

## Summary

Defined a layered input architecture combining the system button, column chords, and encoder-driven interaction to support preset access, editing, and system-level controls.

## Key Updates

- Assigned the onboard system button as the primary **preset control interface**:
  - Tap: toggle between last two presets
  - Hold: open preset picker

- Established **column chords (top+bottom key pairs)** as reserved system-level entry points:
  - Used to enter specific contexts such as system menu, Bluetooth menu, encoder tools, and preset editing

- Defined **preset editor access as contextual**:
  - While in preset picker, pressing the preset editor chord opens the editor for the currently highlighted preset

- Established **encoder as the primary control mechanism** for:
  - Menu navigation
  - Value editing
  - Context-specific adjustments (e.g., sensitivity via hold chord)

- Identified distinction between:
  - Runtime controls (quick access)
  - Editing flows (deliberate)
  - System configuration (rare)

## Architectural Direction

Adopted a 4-tier input model:
1. Direct actions (keys + encoder)
2. Global access (system button)
3. Context entry (reserved chords)
4. Context execution (encoder + keys within modes)

## Notes

- Preset switching remains easily accessible without relying on chords
- Chords are reserved for entering system contexts, not performing actions directly
- Encoder-based adjustments (e.g., sensitivity) will be implemented as temporary hold-chord behaviors initially
- Full menu and edit flows still pending implementation

## Status

- Architecture defined
- Implementation partially complete
- Next steps include:
  - Wiring chord detection into action layer
  - Implementing preset picker behavior
  - Linking encoder configuration to profile changes
